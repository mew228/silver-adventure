## 🔐 How Token Vault is Used (4 Integration Points)

- **Normal delegated OAuth** — User connects Gmail → Vault stores & manages the refresh token. Bridge fetches a fresh access token per request, never cached in process memory.
- **Silent token refresh** — Vault detects tokens near expiry and rotates them using the stored refresh token — no user interaction required, workflow never interrupts.
- **Async authorization** — If a workflow hits an unlinked provider (e.g. Jira, Notion), the bridge suspends the job, sends an auth URL generated by Vault, and auto-resumes after consent.
- **Step-up authentication** — High-risk actions (send email, delete, publish) require explicit user re-consent via Auth0 MFA / step-up before the bridge proceeds, regardless of risk posture.

## ⚡ Quick Start

```bash
git clone https://github.com/mew228/silver-adventure.git

# Copy environment template (app runs in mock mode by default)
cp .env.example .env

```

> **No API keys needed.** The app runs fully end-to-end in mock mode (`MOCK_PROVIDERS=true`).

## 🛠️ Environment Variables

| Variable | Required | Description |
|---|---|---|
| `AUTH0_DOMAIN` | Yes (prod) | Auth0 tenant domain (e.g. `your-tenant.us.auth0.com`) |
| `AUTH0_CLIENT_ID` | Yes (prod) | Auth0 application Client ID |
| `AUTH0_CLIENT_SECRET` | Yes (prod) | Auth0 application Client Secret |
| `AUTH0_AUDIENCE` | Yes (prod) | Token Vault API audience URL |
| `AUTH0_MANAGEMENT_TOKEN` | Yes (prod) | Management API token for Vault operations |
| `AUTH0_CALLBACK_URL` | Yes (prod) | OAuth redirect URI (e.g. `http://localhost:3000/callback`) |
| `ANTHROPIC_API_KEY` | Optional | Claude API key for AI intent decomposition (falls back to keyword planner) |
| `MOCK_PROVIDERS` | Optional | Set `true` to run without real credentials (default: `true`) |
| `PORT` | Optional | Server port (default: `3000`) |
| `NODE_ENV` | Optional | `development` or `production` |

---

## 🔄 Swapping Mock Providers for Real Credentials

1. Set `MOCK_PROVIDERS=false` in your `.env`
2. Fill in the Auth0 variables (see table above)
3. In your Auth0 tenant:
   - Create a new Application (Regular Web App)
   - Enable Social connections: Google, GitHub, Slack, Notion, Atlassian
   - Enable Token Vault for each connection
   - Set Allowed Callback URLs to `http://localhost:3000/callback`
4. Fill in optional provider variables if managing OAuth apps independently
5. Restart: `npm start`

## 🧪 Judges: Testing Token Vault Without Real Credentials

The entire demo runs in mock mode — no Auth0 or provider credentials required:

1. `npm install && npm start`
2. Open `http://localhost:3000`
3. Type a goal: *"Check my emails and create a GitHub issue for action items, then post to Slack"*
4. Select **Gmail**, **GitHub**, **Slack** → posture **MEDIUM** → click **Plan & Execute**
5. Observe the **step-up consent modal** fire for the Slack send action
6. Click **Approve** → watch the audit log confirm execution
7. Now add **Jira** to the selection and re-execute — observe the **async-auth banner** (Jira not pre-connected)
8. Click **Simulate Auth Completion** → job resumes automatically
9. Token Vault references appear in `src/vault/client.ts`, `src/vault/delegated.ts`, `src/vault/async-auth.ts`, and `src/vault/stepup.ts`
