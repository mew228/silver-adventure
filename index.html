Authorized to Act Hackathon prototype

Keep sovereign AI restricted and still let it act in the real world.

Bridgekeeper is a runnable prototype for a two-layer agent system: a restricted local planner plus a governed execution bridge powered by Auth0 for AI Agents Token Vault. The local agent stays away from raw third-party credentials while the bridge handles delegated access, approvals, async auth, and step-up authentication across Gmail, Jira, Notion, Slack, GitHub, and Google Calendar.

"Keep the model local. Keep the action layer governed. Keep the user in control."

System Design

A safer path from local intent to external action.

1. Local Intent Agent

A local OpenClaw-style runtime interprets the user's goals, breaks work into tasks, and requests capabilities. It never stores refresh tokens or runs raw OAuth flows.

2. Bridgekeeper Gateway

The intermediary agent maps intents to approved capabilities, applies policy, triggers approvals, and returns minimized results back to the local model.

3. Auth0 Token Vault

Token Vault manages delegated access, refreshes tokens, supports async auth, and enables step-up authentication for sensitive actions.

Local agent asks for capability
->
Bridgekeeper checks policy + consent
->
Auth0 Token Vault handles auth + token use
->
Scoped action happens in Gmail / Notion / Jira / more

Why it matters

Why this is stronger than giving an agent direct OAuth access.

Restricted-mode friendly

Users can keep local AI in a safer sandbox while still giving it useful, real-world capabilities.

Capability-based permissions

The local model asks for tasks like jira.create_issue instead of broad, long-lived API access.

Human control preserved

Sensitive actions can require approval, re-consent, or step-up authentication before execution.
